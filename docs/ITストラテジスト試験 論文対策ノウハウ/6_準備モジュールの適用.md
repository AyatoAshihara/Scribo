# 1. 準備モジュールの体系と基本戦略
「準備モジュール」とは、どのような設問が来ても対応できるよう、事前に準備しておく論理の塊（パーツ）のことです。

**モジュールの分類**
- 設問ア向け： 背景系（会社紹介、事業概要、課題）
- 設問イ向け： 調査系、分析系、着眼点・観点系、状況報告系、技術活用系、計画内容系
- 設問ウ向け： 対策系（解決策、実行計画）、振り返り系

**執筆の鉄則**
- 直接的な表現で返す： 設問の要求事項が「〇〇について述べよ」であれば、文中で「〇〇は……である。」と明示的に書く。
- 数値と固有名詞の具体性： 「多くの」ではなく「3部門にまたがる80名」のように、採点者がイメージを共有できる具体性を持たせる。
- 矛盾点の埋め込み： 現場の反発やレガシーシステムの制約など、あえて「困難な状況」を記述することで、解決策の論理性を際立たせる。

# 3. 実践：問題文への適応プロセス（一般化のノウハウ）
資料（3.6.2〜3.6.3）にある通り、モジュールをそのまま貼り付けるのではなく、問題文のキーワードに「寄せて」書く技術が必要です。

- キーワードの抽出： 問題文の中にある「デジタル技術」「課題」「解決策」といった単語を特定する。
- モジュールの選択： 準備したパーツ（例：AI、クラウド、RPA）の中から、問題文の文脈に最も近いものを選ぶ。
- 接続詞での調整： 「なぜならば」「具体的には」「一方、現場では」といった接続詞を用いて、自社の背景と解決策を論理的に繋ぐ。

# 4. AI論文生成用「統合指示書（プロンプト）」
以下の内容をLLMに与えることで、提供資料のノウハウに基づいた論文生成が可能になります。

**システムプロンプト構成案**
>
# 役割
あなたはITストラテジスト試験の合格論文執筆パートナーです。提供された「準備モジュール」を駆使して論理的な論文を構成します。

# 執筆ルール
1. **設問への直接回答**: 設問ア、イ、ウの各要求に対し、「〇〇については以下の通りである」と宣言してから内容を記述すること。
2. **定量的具体性**: 従業員数、削減コスト、処理件数などの数値は「〇〇」とせず、現実的な数値（例：15%削減、3ヶ月で導入）を補完すること。
3. **矛盾と解決**: 成功体験だけでなく、必ず「現場の反発」や「技術的制約」という障害を記述し、それをどう乗り越えたかの論理を示すこと。
4. **モジュールの埋め込み**: 以下のフレーズを適宜文中に組み込むこと。
   - 「優先度の高い部分から段階的に……」
   - 「パイロットフェーズで効果を測定してから……」
   - 「企業統合による業務プロセスの混在が……」

# 論文構成の指示
- 設問ア：800字以内（背景、事業特性、課題）
- 設問イ：800〜1600字（分析、IT戦略、検討プロセス）
- 設問ウ：600〜1200字（実行計画、評価、今後の課題）
