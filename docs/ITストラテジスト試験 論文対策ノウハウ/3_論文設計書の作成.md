# 1. 論文設計書作成の核心ノウハウ
論文をいきなり書き始めるのではなく、**「設計書（章・節の構成案）」**をまず作ることが合格への最短ルートです。

**執筆の基本ルール**
- 設問分解の徹底: 設問文（ア、イ、ウ）を、さらに細かい「要求事項」に分解する。
- 章・節タイトルの決定: 分解した要求事項を組み合わせて、採点者が一目で「要求に応えている」と分かるタイトル（例：2.1 ○○、2.2 △△）を付ける。
- 準備モジュールの活用: 過去の経験や汎用的なフレーズ（モジュール）を、設問のキーワード（「分析」「解決策」など）に合わせて当てはめる。
- 数値と固有名詞の具体性: 「情報の鮮度が悪い」「手作業が残っていた」といった現状に対し、「(s385)情報の鮮度が悪い」といった管理番号や具体的な状況を紐付ける。

# 2. 論文設計書の論理構成（テンプレート構造）
資料に基づいた標準的な構成案です。

第1章：事業概要と事業特性、分析対象業務とITシステム
1.1 事業概要と事業特性: 会社紹介、事業規模、業界特性。
1.2 利用部門からの改善要望内容と問題認識: 現状の課題（「情報のリアルタイム共有の欠如」など）。

第2章：収集した情報と分析方法、特定した問題の真因と工夫したこと
2.1 収集した情報と分析方法: インタビュー、現場調査、データ分析（パレート図、ABC分析等）。
2.2 特定した問題の真因: 個別最適による弊害、手作業による遅延など。
2.3 工夫したこと: 現場の本音を引き出すためのインタビューの工夫、多角的な分析。

第3章：利用部門や関係部門との協議、対応方針
3.1 利用部門や関係部門との協議: 合意形成のプロセス。
3.2 問題の真因に対し立案した対応方針: RPAの導入、システムの刷新、業務プロセスの変更など。

# 3. 具体的な記述例（Few-shot用サンプル）
設問ごとの構成および、そのまま使える具体フレーズです。

[設問ア] 事業概要・現状分析
- 「(1030) C社は〇〇〇を専門とする中堅の〇〇〇会社である。」
- 「(1275) 発言力の強い一部の部門が、要望を乱発する傾向があった。」
- 「アクセス→営業支援システムの在庫情報の内容を常に最新にしてもらいたい」

[設問イ] 問題の特定・分析
- 分析系モジュール: 「親和図を作成し、関係の深い事項をグループ化し、分析した。」「パレート図を作成し、最も効果の上がる部分を抽出・定量化し、分析した。」
- 真因の特定: 「個別最適の視点だけで検討した部分的な内容にとどまっている可能性がある」「手作業が残っていた」「在庫切れ、売り逃しが発生」

[設問ウ] 解決策・工夫点
- 対策系モジュール: 「集合教育（啓蒙教育）を行い、一般ユーザの認識を高めた。」「PDCAサイクルが回るよう、年間のスケジュールにイベントとして組み込んだ。」
- 工夫点の表現: 「CIOに問題の真因を掘り下げる必要性があることを説明→正式に許可を得てから関連部門へヒアリングを実施」「3部門関係者を招集→問題の真因の解消に寄与する解決手段について協議」

# 4. 論文生成用：統合システムプロンプト（指示書）
AIに論文を生成させる際、以下のプロンプトをベースにしてください。

【システムプロンプト案】
# 役割
あなたはITストラテジスト試験の合格論文執筆パートナーです。

# 執筆方針
1. **設問への忠実な回答**: 設問ア・イ・ウで問われている項目を、必ず章・節のタイトルに反映させてください。
2. **定量的・具体的な記述**: 「効率化した」ではなく「残業時間を月30時間削減した」など、数値や具体的な管理番号（s445等）を用いてリアリティを出してください。
3. **論理的整合性**: 第1章で挙げた課題が、第2章の分析で真因特定され、第3章の対策で解決されるストーリーを厳守してください。
4. **準備モジュールの組み込み**: 
   - 分析手法（パレート図、ABC分析、ユーザニーズ分析）
   - 対策手法（PDCA化、マニュアル整備、集合教育）
   を適切に選択して具体的に記述してください。

# 構成案の出力形式
## 論文設計書
- 第1章...
- 第2章...
- 第3章...
（各節には、どのような具体的エピソードを盛り込むかの「メモ」を付記すること）
