# 1. IT論述試験対策：背景（ネタ）の準備ガイド

## 1.1 背景（設問ア）の重要性
「背景」は論文の土台であり、採点者がその後の論理構成や対策の妥当性を判断するための重要な基準となります。

- **キャンバスの役割**: 背景が不明瞭だと、その上に描かれる論理（設問イ・ウ）が浮いてしまい、一貫性が失われる。
- **第三者への配慮**: 採点者はあなたの業界の専門家とは限りません。「誰が読んでも状況がイメージできる」記述が不可欠です。

## 1.2 執筆の基本ルールと注意点
- **専門用語・略語の制限**: 業界用語や社内用語を多用しない。使う場合は必ず補足説明を入れる。
- **具体性の確保**: 「ビッグデータ」「DX」といったバズワードだけで済ませず、自社における具体的な定義や課題に落とし込む。
- **制約条件の明示**: 「売上第一主義」「交通系なら安全重視」など、その業界特有の文化や制約を明示すると、後の対策に説得力が出る。

# 2. 論文構成のコンポーネント化（3つのモジュール）
論文の背景は、以下の3つの要素を1セットとして準備しておくことが推奨されています。

| モジュール名 | 内容のポイント | 具体的な記述項目 |
| :--- | :--- | :--- |
| ①会社紹介 | どのような立ち位置の会社か | 事業概要、事業特性、業界内での位置づけ |
| ②現状モジュール | 今、何が起きているか | 事業目標、継続・取組み状況、現在の業務フロー |
| ③不安材料（課題） | 何が問題（リスク）か | 市場変化、競合動向、システム上の限界、現場の不満 |

# 3. 具体的な記述例（Few-shotサンプル）
これらは論文の「設問ア（背景）」にそのまま、あるいは微調整して転用できるテンプレートです。

### ケースA：ビル設備管理会社（現場の抵抗）
| 項目 | 内容 |
| :--- | :--- |
| ①会社紹介 | A社は、某大手建設会社グループに属しているビル設備管理会社である。 |
| ②現状 | 長年グループ内からの仕事を引き受けていたが、近年は異業種からの参入もあり、営業実績が悪化の一途をたどっている。生き残りをかけた打開策として、大規模な業務改革を計画している。例えば、現状ビルごとに異なっている報告書書式や管理システムを統合し、サービスの向上を図りたいと考えている。 |
| ③不安材料 | 現場からは、ベテランを中心に慣れ親しんだシステムの再構築に対する反発が予想された。 |

### ケースB：電力会社（DX推進・新たなビジネスモデル）
| 項目 | 内容 |
| :--- | :--- |
| ①会社紹介 | D社は、大手電力会社傘下のシステム会社である。 |
| ②現状 | 電力の自由化に伴い、同業他社との顧客獲得競争が激化している。E社は、長年契約を継続している高齢顧客が多い。そこで、「高齢者向けの自宅見守りサービス」に着目し、従来になかった新たなビジネスモデルにしていきたいと考えている。家電メーカでは、各電気製品にIoT装置を内蔵したり内蔵させて、高齢者の製品ごとの使用状況を把握できるようにしている。E社では、他社製品に手を加えずに、簡単な設置作業で見守りが開始できるようにしたいと考えている。 |
| ③不安材料 | 顧客宅内にある分電盤に、センサー装置を取り付ける方式がある。しかし、使用状況を把握するには、家電製品ごとの電流パターンを、クラウドが識別できなければならなかった。 |

### ケースC：銀行子会社（システム統合に対する不安）
| 項目 | 内容 |
| :--- | :--- |
| ①会社紹介 | B社は大手銀行のグループに属する〇〇〇会社である。 |
| ②現状 | 業界再編の波を受け、他の銀行や金融機関との統合を長年繰り返してきている。システムは従前どおり、並列に存続しており、業務プロセスも煩雑化している。折しも、監督官庁からの指導があり、強固なシステムの再構築を目指すこととなった。 |
| ③不安材料 | システムを統合した場合、子会社や取引先を含めて影響度は計り知れない。経営層からは、安定した切り替えに対する不安の声が上がっている。 |

# 4. AI論文生成用プロンプト（統合ガイド）
この資料の知見をAIに指示として与える場合、以下のプロンプト形式が有効です。

```markdown
# 役割
あなたは高度IT試験（ITストラテジスト等）の論文執筆エキスパートです。以下の制約に従い、合格水準の「設問ア（背景）」を生成してください。

# 執筆構成案（コンポーネント・モジュール法）
1. 【会社紹介】: 業種、事業規模、市場での立ち位置を具体的に記述（例：〇〇業界シェア3位の製造業など）。
2. 【現状】: 現在取り組んでいるITプロジェクトや、達成すべき事業目標を記述。
3. 【不安材料/課題】: プロジェクトを阻害する要因、現場の抵抗、技術的困難、または外部環境の変化によるリスクを1つ以上明示すること。

# 制約事項
- 業界外の採点者が理解できるよう、社内用語や過度な略語は避け、必要に応じて補足説明を入れる。
- 以下の「準専門用語」を適切に使用し、論理的な文体（だ・である調）で記述すること。
  （例：アーキテクチャ、アライアンス、意思決定支援、依存、委託）
- 具体的な数値（従業員数、売上規模、削減目標時間など）を盛り込み、リアリティを出すこと。

# 入力情報（ネタの種）
[ここに具体的な案件の内容や、上記「3. 具体的な記述例」のどのケースに合わせるかを記入]
```
